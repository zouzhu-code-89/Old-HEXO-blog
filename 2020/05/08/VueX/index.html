VueX 全局状态管理教程

2020-05-08

Vuex 是一个专门为 Vue.js 应用程序开发的 状态管理模式 ，它采用集中式存储管理应用的所有组件的状态，并以相应的规则保证状态以一种可预测的方式发生变化，Vuex 也集成到 Vue 的官网调试工具 devtools extension，提供了诸如零配置的 time-travel 调式、状态快照导入导出等高级调试功能.

我们为什么要使用 "状态管理" 呢 ?
因为我们在组件中定义的数据只在当前组件中有效，一个大型项目中有成百上千的组件或组件构成的页面，如果我们离开当前组件，数据也将在一次刷新，所以如果我们想保存用户的状态行为，我们就必须要声明一个全局的状态管理器，这个全局状态管理器将应用到整个项目的上下文，所以你开发大型项目时一定会考虑使用这个技术，同时你也要考虑什么数据，是你将要声明为全局的状态，否则大量的全局状态会造成数据污染，之后我们会对数据污染进行讲解.

VueX 状态管理模式

Vuex 是一个专为 vue.js 应用程序开发的状态管理模式，它采取集中存储管理应用的所有状态 ，并以相应的规则保证状态以一种可控的方式发生改变 。

#什么是 Vuex 状态管理模式 ?

状态管理应用包含以下几个概念 :

  • state ：vuex 驱动应用的数据源 。
  • view ：以声明方式将 state 映射到视图 。
  • actions：响应 view 上，用户输入导致的状态变化 。

以下是一个表示单向数据流理念的简单示意 :

vuex 中定义的数据源(类似 vue data)，会在 view template 组件中展示，当用户在 view 上改变数据源的数据时，action 会响应到这个数据的变化，这个响应不附带监听效果，而是需要我们主动调用 vuex 的提交，响应后会改变 state 数据的状态 ；也许你现在不懂 state ，但是你阅读下去就会懂这一段话了 !

vuex 使用问题分析 :

但是当我们的应用遇到多个组件共享状态时，单向数据流的简洁性很容易被破坏：

  • 多个视图依赖于同一状态。
  • 来自不同视图的行为变更同一状态。

store 实例只能有一个，如果我们在多个组件文件中通过调用 store 设置数据源，随意使用这个 state 全局状态，就很容易导致上面的数据混乱问题 。

解决方案 :

  • 我们应该把组件所有的共享状态抽取出来，以一个全局单例模式管理(也就是将store封装成一个模块”单文件形式”，统一管理)，这样更利于开发人员维护状态 。

  • vuex 是专门为 vue.js 设计的状态管理库，以利用 vue.js 的细粒度数据响应机制来进行高效的状态更新。

#什么情况下我应该使用 Vuex ？

vuex 可以帮助我们管理共享状态，如果你不算开发大型单页应用，使用 Vuex 可能是繁琐冗余的，确实如此–如果你的应用够简单，你最好不要使用 vuex，一个简单的 store 模式就足够您所需了 。

但是如果您需要构建一个中大型单页应用，你很可能会考虑如何更好地在组件外部管理状态，vuex 将会成为自然而然地选择 。

State 数据源

每一个 vuex 应用的核心就是 store（仓库），”store” 基本上就是一个容器，它包含着你的应用中大部分的状态数据 .

state 和 data 有以下两点不同 :

  • vuex 的状态数据存储是响应式的，当 vue 组件从 store 中读取状态的时候，若 store 中的状态发生变化，那么相应的组件也会相应地得到高效更新 。
  • 你不能直接改变 store 中的状态，改变 store 中的状态的唯一途径就是显式地提交 (commit) mutation。这样使得我们可以方便地跟踪每一个状态的变化，因为 mutation 里面的函数都是我们自己定义的，所以改变是可控制、可追踪的 。

4.state 总结

单一状态树

vuex 使用单一状态树、用一个对象就包含了全部的应用组件层级状态、至此它便作为一个”唯一数据源” ssot 而存在、这也意味着每个应用(项目)将仅仅包含一个 store 实例、单一状态树让我们能够直接地定位任一特定的状态片段、在调试的过程中也能轻易地取得整个当前应用状态的快照 。

单状态树和模块化并不冲突、存储在 vuex 中的数据和 vue 实例中的 data 遵循相同的规则、例如状态对象必须是纯粹的 。

使用 Vuex 并不意味着你需要将所有的状态放入 Vuex、虽然将所有的状态放到 Vuex 会使状态变化更显式和易调试，但也会使代码变得冗长和不直观。如果有些状态严格属于单个组件，最好还是作为组件的局部状态。你应该根据你的应用开发需要进行权衡和确定。

Mutation 改变状态(同步)

Store 里面的 state 状态是不能够改变的、就是说如果你强行改变了 state 的状态、控制台是会报错的、state 也不会随之改变、所以这样做是没有意义的 。Vuex 给我们提供的方案就是 mutations 属性、它里面包含我们定义的一些事件、这些事件就是对状态的操作、且我们定义多少事件、就有多少种改变状态的方案 。

这样的目的就是让 state 状态改变变的更加可控制、开发者不能随意改变 state 里面的状态、而是需要通过 mutation 里面的事件、我们需求是怎么样的、就在里面定义什么样流程类型的事件 。

你不能直接调用一个 store mutations 里面的事件、这个选项更像是事件注册、当你想要调用里面的事件时，你需要使用 store.commit(‘increment’) 提交方法、实例会帮你去调用这个方法 。

3.同步带来的问题 ？

在 mutation 中混合异步调用会导致你的程序很难调试。例如当你调用了两个包含异步回调的 mutation 来改变状态，你怎么知道什么时候回调和哪个先回调呢？这就是为什么我们要区分这两个概念。在 Vuex 中、mutation 都是同步事务、为了处理异步操作、让我们来看一看 Action

Action 改变状态(异步)

action 类似于 mutation 不同在于下面的两点

  • action 提交的是 mutation，而不是直接变更状态、事件内部还是 commit
  • action 可以包含任意异步操作。

上面两点我们可以知道 action 并不具备改变 state 状态的能力、只是在 mutation 的基础上添加异步操作、主要就是解决 mutation 只能同步操作的局限性 . 事件内部还是需要提交 commit mutation 里面的事件、所以没有 mutations 、actions 将发挥不了任何作用 .

9.保留 state

在注册一个新 module 时，你很有可能想保留过去的 state，例如从一个服务端渲染的应用保留 state。你可以通过 preserveState 选项将其归档：store.registerModule('a', module, { preserveState: true })。

当你设置 preserveState: true 时，该模块会被注册，action、mutation 和 getter 会被添加到 store 中，但是 state 不会。这里假设 store 的 state 已经包含了这个 module 的 state 并且你不希望将其覆写。

10.模块重用

有时我们可能需要创建一个模块的多个实例，例如：

  • 在一个 store 中多次注册同一个模块

项目开发结构

Vuex 并不限制你的代码结构、但是它规定了一些需要遵守的规则 :

  1. 应用层级的状态应该集中到单个 store 对象中。
  2. 提交 mutation 是更改状态的唯一方法，并且这个过程是同步的。
  3. 异步逻辑都应该封装到 action 里面。

只要你遵守以上规则，如何组织代码随你便。如果你的 store 文件太大，只需将 action、mutation 和 getter 分割到单独的文件。
